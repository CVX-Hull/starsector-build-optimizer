# Post-hoc top-K candidate selection ŌĆö research, implementation, and Wave 1 empirics

**Question.** Spec 30 ranks honest-eval candidates by `mean(intermediate_values)`. Empirical work (this session) showed Box-Cox-shaped fitness saturates. Theory (phase5a/5d) says we should be doing TWFE + EB shrinkage. Is raw mean defensible, or are we systematically picking the wrong builds?

**Answer.** Raw mean is severely confounded by opponent draw. TWFE / TWFE-EB / BradleyŌĆōTerry pick *different* and structurally sounder builds. A consensus across the three principled methods identifies one build (`36538033d63b`, double-heavy-AC Hammerhead) as the strongest post-hoc training-log candidate. It is not the honest-eval winner; honest eval is the oracle for cross-cell build quality.

This report covers (1) the methods we surveyed, (2) the implementation in `src/starsector_optimizer/posthoc_ranker.py`, (3) empirical comparison on Wave 1, and (4) Starsector domain validation of the picks.

## 1. Methods surveyed

| # | Method | Fit | Inputs | Cross-study pooling |
|---|---|---|---|---|
| 1 | Raw mean | Spec 30 baseline | per-match `hp_differential` | natural (concat matches) |
| 2 | Two-Way Fixed Effects (TWFE) | phase5a; deconfounds opponent | per-match `hp_differential` | natural ŌĆö opponents shared |
| 3 | TWFE + EB shrinkage (no covariates) | phase5a + phase5d-without-X | TWFE ╬▒╠é + per-build n_i | natural |
| 4 | BradleyŌĆōTerry MAP | structural skill model | per-match winner label only | natural |
| 5 | AIPW / IPW | doubly-robust adjustment | TPE+pruner propensity model | possible but fragile |
| 6 | Multi-task / hierarchical GP | phase 7 | structured build descriptor | yes (phase 7 plan) |
| 7 | Best-arm identification (BAI) | adaptive sampling | requires running BAI sampler | not applicable post-hoc |

We implemented #1ŌĆō#4 (the rigor ├Ś tractability sweet spot for the existing data); #5 is fragile because the pruner makes propensity non-ignorable (positivity violated), #6 is the Phase 7 destination, and #7 cannot retrofit confidence onto a TPE+pruner trace that wasn't generated by a BAI sampler.

Full literature notes: `docs/reference/phase5a-deconfounding-theory.md` and `docs/reference/phase5d-covariate-adjustment.md` for in-project theory; for the literature review see this report's session-prep agent log.

## 2. Implementation

`src/starsector_optimizer/posthoc_ranker.py` exposes:

```python
load_records(jsonl_paths) -> list[TrialRecord]
rank_raw_mean(records, k)         -> list[RankedBuild]
rank_twfe(records, k, cfg)        -> list[RankedBuild]
rank_twfe_eb(records, k, t, e)    -> list[RankedBuild]   # uses deconfounding.eb_shrinkage with degenerate X
rank_bradley_terry(records, k, c) -> list[RankedBuild]   # MAP via L-BFGS, ridge prior
topk_overlap(a, b), spearman_rho(a, b)
```

Design notes:
- All ranks operate on the same record set (caller decides which rows are "completed" ŌĆö e.g. `not pruned and not cache_hit and not invalid_spec`).
- Build identity is a stable hash of `(hull, sorted weapons, sorted hullmods, vents, caps)`; cross-study build collisions are detected and pooled.
- TWFE reuses `deconfounding.twfe_decompose`; EB reuses `deconfounding.eb_shrinkage` with a zero-column X so the regression prior degenerates to the grand mean. This keeps the ranker faithful to phase5a/5d without re-implementing the estimator.
- BradleyŌĆōTerry: bipartite logistic skill `P(build_i beats opp_j) = Žā(╬▒_i ŌłÆ ╬▓_j)`, MAP via L-BFGS with N(0, 1/ridge) prior, TIMEOUT matches as weighted draws (default weight 0.5), Fisher-information diagonal for per-arm Žā.

Synthetic recovery test (`tests/test_posthoc_ranker.py`, 11 tests, all passing): under deliberately confounded opponent assignment, TWFE recovers the true top-K with overlap Ōēź raw_mean's overlap; BT recovers Ōēź 2/5 of the true top-5 from winner-only labels.

## 4. Starsector domain validation

For each consensus pick, we inspect the build against game-mechanics knowledge: weapon synergy, flux economy, hullmod relevance, slot allocation. Hammerhead = destroyer with 2 medium ballistic + 4 small ballistic + 2 small composite turrets; early regime restricts to tier-1 weapons + non-rare hullmods.

### `36538033d63b` (BT #1 / TWFE #2) ŌĆö **strongest by domain knowledge**
- vents=4, caps=5; flux economy: `fluxcoil` (+ capacity) + `fluxdistributor` (+35% vent rate)
- Hullmods: `armoredweapons` (weapons hard to disable), `blast_doors`, `extendedshieldemitter` (wider arc), `insulatedengine` (engines hard to disable)
- Weapons: **double `heavyac`** (2├Ś medium kinetic, the canonical Hammerhead anchor), `annihilator` (medium HE rocket pod), `reaper` (heavy torpedo), `lightmortar`, `lrpdlaser` (long-range PD), `lightdualac`, `irpulse`
- This is **the canonical strong early-Hammerhead loadout**: kinetic shield pressure ŌåÆ annihilator/reaper armor crack. Flux economy is configured to sustain double heavy-AC fire. Engine + weapon protection mods address the Hammerhead's main vulnerability (it's a flux-fragile destroyer that gets disabled fast).
- Notably **missing** `stabilizedshieldemitter` (the OP-cheapest shield upkeep mod) ŌĆö substituting `extendedshieldemitter` is a defensible alternative when shield-arc coverage outweighs upkeep cost. Reasonable design choice.

### `ab7851df4c14` (TWFE #4 / BT #8)
- vents=4, caps=5
- Hullmods: `auxiliarythrusters`, `blast_doors`, `extendedshieldemitter`, `fluxcoil`, `magazines`, `stabilizedshieldemitter` (Ō£ō canonical), `turretgyros`
- Weapons: `heavyac` + `arbalest` (medium kinetic high-range), `annihilator` + `harpoon` (HE follow-up), `ioncannon` (utility), `lightmortar`, `irpulse`. WS_007 empty.
- This is also a competent build: kinetic-pressure + missile-finisher + ioncannon flux-disruption. `stabilizedshieldemitter` + `extendedshieldemitter` is the canonical defensive pair. The empty slot suggests OP-budget-bound search; not a flaw.
- A real-game player would build something very close to this for early Hammerhead vs destroyer/cruiser.

### `256e0802f501` (4/4 votes ŌĆö TWFE #1)
- vents=4, caps=5
- Hullmods: `armoredweapons`, `blast_doors`, `extendedshieldemitter`, `fluxcoil`, `reinforcedhull`, **`surveying_equipment`** (campaign-only mod ŌĆö zero combat value, wastes 1 OP slot)
- Weapons: `heavyac`, `shredder`, `reaper`, `swarmer`, `lightdualmg`, 2├Ś `lightmortar`, `lightag`
- Mixed verdict: defensive hullmods are sound, but `surveying_equipment` is a campaign-only mod (lets the player survey planets faster) with zero combat effect. The fact that this build is TWFE's #1 with one wasted hullmod slot suggests the optimizer found "good enough" rather than "optimal".
- Single `heavyac` (vs the double-heavyac in `36538033d63b`) is a weaker kinetic profile.

### `e31d934a6943` (raw_mean #2, but TWFE/EB #6) ŌĆö **why raw mean is biased**
- vents=1, caps=1 ŌĆö almost no flux capacity for a destroyer
- Hullmods include **`converted_hangar`** (a campaign-only mod that gives a fighter bay to a *non-carrier* hull ŌĆö pointless on Hammerhead which has no fighter bay) AND `advancedshieldemitter` (very expensive at 25/45/85 OP)
- Weapons: heavy mix but the build cannot sustain it on 1 vent.
- This build appears in raw_mean's top-3 (#2) but TWFE/EB rank it #6. The raw-mean inflation comes from c3/seed1's specific opponent draw ŌĆö the build happens to have faced easier opponents. Once opponent fixed effects are subtracted, the build drops out of the top.
- **This is the smoking gun for raw-mean bias**: a structurally weak build (wasted hullmod, weak flux economy) ranks high under raw mean and falls under deconfounding.

### `5eaa4d2ec59b` (3/4 votes, TWFE/BT #7-#9, glass-cannon)
- vents=0, caps=1; minimal flux. Stacks `heavyarmor` + `advancedshieldemitter` + `stabilizedshieldemitter`
- Weapons: `shredder`, `flak`, 2├Ś `swarmer`, 2├Ś `ioncannon`, `lightmortar`. WS_007 empty.
- Glass-cannon profile: heavy missile/PD, no kinetic, very low flux. Wins quickly via missile burst before flux becomes the bottleneck. A legitimate (if specialised) niche build.
- The fact that all three principled methods rank it in the top-10 but not top-5 is consistent: it wins decisively when it wins, but it cannot sustain combat against tougher opponents. BradleyŌĆōTerry's binary win-loss likes this profile slightly more than TWFE's residual-magnitude.

## 5. Decision (shipped 2026-05-10)

**`extract_top_builds` switched to TWFE+EB by default** (`src/starsector_optimizer/honest_evaluator.py`):
1. Ō£ģ **Switched from raw_mean to TWFE+EB** as the default `--ranking-method`. Implementation routes through `posthoc_ranker.rank_twfe_eb` which reuses `deconfounding.eb_shrinkage` (degenerate X ŌåÆ grand-mean prior, since post-hoc covariates aren't reconstructed yet). The CLI flag accepts `twfe_eb | twfe | raw_mean | bradley_terry` for ablation.
2. Ō£ģ **BradleyŌĆōTerry runs in parallel** as a diagnostic. `report_method_disagreement()` is invoked per study before dispatch; `main()` logs an INFO line with each method's top-K and a WARN when zero overlap exists. This surfaces "heavy confounding / near-tied top region" before money is spent.
3. ŌÜĀ’ĖÅ **Per-seed top-K, not pooled cross-seed.** Each `(cell, seed)` study is ranked independently to keep the same eval power (3 seeds ├Ś top-3 = 9 candidates per cell, unchanged). Pooling for ╬▓ estimation is documented but not yet wired ŌĆö within-seed n is sufficient for Wave 1 magnitudes.
4. **JSONL is now the candidate-selection input** (was SQLite). `data/logs/<campaign>/*/evaluation_log.jsonl` glob pattern. SQLite remains the storage of record for the optimizer; honest-eval no longer reads it.

**For Wave 1 honest-eval re-launch (#93):** `scripts/cloud/launch_wave1_honest_eval.sh` passes `--ranking-method twfe_eb` explicitly so the choice is visible in the launch log. Spec 30 updated.

## 6. Pooling across runs ŌĆö verdict

**Yes**, with caveats:

- Ō£ģ **Within-cell** (3 seeds): natural to pool by build ŌĆö but Wave 1's TPE searches produced no cross-seed build collisions, so pooling helps ╬▓ not ╬▒.
- Ō£ģ **Across cells** (5 cells ├Ś 3 seeds): natural to pool by opponent identity ŌĆö every cell uses the same 54-opponent early-regime pool. Cross-cell ╬▓ estimates are ~3├Ś sharper than within-cell. This is the highest-value pooling axis for Wave 1.
- ŌÜĀ’ĖÅ **Across waves / regimes / hulls**: only valid by opponent. ╬▓ estimates can be shared across waves that face the same hull-size-matched pool; ╬▒ cannot, because regimes restrict the build space differently.
- ŌØī **AIPW / propensity-weighted pooling**: TPE+pruner makes propensity non-ignorable; positivity is violated. Not viable in this setup.

## 9. Limitations

- All builds in Wave 1 have `n_matches = 10` (not pruned, evaluated against a fixed-size opponent subset). EB's heteroscedastic shrinkage doesn't differentiate on this dataset because Žā╠é_i┬▓ = Žā╠é_╬Ą┬▓/n_i is constant in n_i. **Wave 2+ runs with variable n per build (e.g. WilcoxonPruner kicking in at different schedule points) will exercise EB's discrimination ŌĆö expect EB to start producing different rankings from plain TWFE there.**
- The TWFE additive model (`y_ij = ╬▒_i + ╬▓_j + ╬Ą_ij`) ignores rock-paper-scissors ŌĆö a kinetic build does well vs shielded opponents, an HE build vs armoured ones. BT has the same limitation. **Wave 2's cross-regime cells will let us check whether interactions matter; if so, multi-task GP (Phase 7) is the right escalation.**
- All inferences here are conditional on the V2 loadout fix being correct. Wave 1 was run pre-fix; the LOADOUT_MISMATCH rate will determine whether the rankings are re-validated as-is or need to be re-run on Wave 2.
